Master Backend Audit Report

Authentication, SSO, SAML & SCIM Architecture

Executive Summary

This report documents a comprehensive architectural and security audit of the backend authentication module. The system utilizes Fastify, Zod for schema validation, and monolithic service patterns (`service.ts` at 106KB). While it possesses advanced enterprise capabilities (SCIM, SAML, WebAuthn, TOTP), it suffers from tight coupling and potential architectural bottlenecks.

1. Authentication Flow Architecture

The system relies on an API-driven Fastify layer isolating validation (`types.ts`) from business logic (`service.ts`) and data access (`repository.ts`).

Email/Password Login

  • Entry Point: POST /auth/login
  • Validation: LoginSchema (Zod) ensures valid payload format.
  • Service: loginWithEmailPassword() computes bcrypt hash, validates against DB.
  • MFA Intercept: Returns 202 Accepted with mfa_required: true if TOTP/WebAuthn is enrolled.
  • Session Creation: Generates Access Token (JWT) and Refresh Token (Opaque/Signed).

2. Session Architecture

The system operates entirely Redis-free for session persistence. Sessions rely strictly on PostgreSQL for cross-process state, with an in-process LRU cache (cache.ts) handling real-time revocations and challenge states.

  • Refresh Tokens: Stored as an HttpOnly, Secure cookie. Production uses __Host-refresh_token with Path=/ to prevent subdomain hijacking.
  • Rotation: Refresh tokens are rotated upon use, with a short retry-grace window to prevent network race condition lockouts.
  • Revocation: Global revocations flag sessions in DB, immediately invalidating them via the LRU cache propagation.

3. JWT Audit

  • Strategy: Short-lived Access Tokens (Bearer) passed in memory, validated statelessly via signature.
  • Claims: Standard sub, exp, iat, jti.
  • Critique: Zero tenant-data (organization ID, roles) is stored in the JWT to prevent stale authorization logic. Organization mapping is strictly retrieved dynamically at egress points.

4. SSO & SAML Audit

SSO relies on dedicated routing layers (saml-identity.routes.ts, sso-oidc.routes.ts).

  • SAML Flow: Implements SP-initiated flows. Relies on saml-xml.util.ts for XML parsing. Requires strict validation of InResponseTo to prevent replay attacks.
  • OIDC Flow: Follows OAuth2 Authorization Code flow with PKCE. State parameter validates CSRF.
  • JIT Provisioning: Users are Just-In-Time provisioned into default groups based on IdP attributes.

5. SCIM Audit

Recently patched to resolve a PostgreSQL schema mismatch with Zod enums.

  • Auth: Secured via scoped long-lived Bearer tokens (organization_scim_tokens).
  • Scopes: users:read, users:write, users:delete, groups:read, groups:write, groups:delete, bulk.
  • Endpoints: Standard /Users and /Groups.

6. Security Audit (Ranked Risks)

P1 Risk (High): Monolithic Service Coupling
src/modules/auth/service.ts is over 100KB (3,500+ lines). This creates massive merge conflict risks, testing difficulty, and hidden circular dependencies. Evidence: Single file containing user lifecycle, TOTP, WebAuthn, password resets, and session management.
P2 Risk (Medium): In-Process LRU Limitation
Relying on cache.ts for revocation states means clustered deployments (Kubernetes multi-pod) will suffer from split-brain revocation delays until the DB polls sync.

8. Code Quality & Refactoring Recommendations

Current State Recommended Action Reason
service.ts (106KB) Split into session.service.ts, mfa.service.ts, password.service.ts. Separation of concerns, testability.
repository.ts (80KB) Implement Domain-Driven Repositories (e.g., user.repository.ts, session.repository.ts). Reduces cognitive load and transaction scope complexity.
In-Process LRU (cache.ts) Migrate to Redis for clustered enterprise deployments. Prevents split-brain session revocations across multiple pods.

9. Phased Migration Plan

  1. Phase 1: Safe Extraction (No API changes) - Extract MFA logic (TOTP/WebAuthn) from service.ts into a dedicated mfa.service.ts. Update imports.
  2. Phase 2: Repository Splitting - Break repository.ts into entity-specific data layers. Keep database transactions localized.
  3. Phase 3: Redis Integration - Swap cache.ts LRU memory map for an IORedis connection block for instantaneous global revocations.
