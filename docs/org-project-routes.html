Executive Summary

The Organization module should own tenant-level concerns. The Project module should own project-level concerns. The critical finding is an API-key overlap that must be resolved before this becomes a public contract.

✓ Core Conclusion

Organization routes manage the tenant. Project routes manage monitored applications. Project API Keys are the only ingestion keys. Organization-level keys should be removed, hidden, or renamed to service tokens.

🏢 Organization module owns

Membership, access policy, SSO, SCIM, audit logs, security events, environments, and quota governance.

📂 Project module owns

Monitored applications, project lifecycle, project status, and project-scoped ingestion API keys.

Enterprise Domain Model

The clean product boundary that drives all route ownership decisions. Tenant-level authorization lives in the Organization; application workspaces live inside Projects.

Risk & Gap Assessment

Current known risks ordered by severity. Critical items block production launch. High items should be resolved for enterprise readiness.

⚠ Risk Register
RiskSeverityImpactRecommendation
Project API-key access checks are commented out in ProjectsService. Critical Users may list or create project keys without proper project permission checks. Re-enable requireProjectAccess for list/create before production.
Debug logs exist in project routes for request body and API-key creation. High Sensitive payloads may appear in logs. Remove console.log("request.body",...) and console.log("apikey",...).
Organization API keys overlap with project API keys. High Product confusion and security review complexity. Deprecate or rename org keys to service tokens.
Organization key rotation uses fixed/default values in service implementation. Medium Weak operational behavior and unclear metadata after rotation. Require explicit name/role/env/expiry or preserve existing metadata.
Public utility routes may allow enumeration. Medium Slug or invitation validation routes can leak availability patterns. Add rate limits and generic responses where appropriate.
Enterprise features present before complete enforcement exists. Medium UI/API may imply SSO/SCIM/security maturity before end-to-end enforcement. Gate exposure behind product readiness and tests.

Implementation Roadmap

Three phases from security stabilization through enterprise readiness. Phase 1 items are blocking for production launch.

1

Phase 1 — Security Stabilization

  • Re-enable permission checks for project API-key list/create
  • Remove debug logging from project routes
  • Add focused tests for all P0 routes (org create/list/get, member role update/removal, project create/get/update, project API-key create/list/disable/rotate/delete)
  • Verify every mutating P0/P1 route writes audit records
2

Phase 2 — API Surface Cleanup

  • Decide whether organization-level API keys are needed
  • If not needed, remove from public docs and Postman collections
  • If needed, rename to service tokens or management tokens
  • Add scopes, usage, last-used tracking, enable/disable, metadata reads, and clear audit events for service tokens
3

Phase 3 — Enterprise Readiness

  • Harden SSO and SCIM flows with token/certificate handling rules
  • Add rate limits for public validation routes (slug-available, invitation validate)
  • Add export format and date filtering to audit logs
  • Add manager/admin-facing documentation for roles, permissions, and route ownership
